# GenLab Launchpad LMS API - Credentials & Database Guide

This document explains how the Spring Boot backend connects to the database in different scenarios and how to manage credentials securely using AWS Secrets Manager.

---

## 1. Overview & Architecture

The application is configured to run in two distinct database connection scenarios:
* **Scenario A (Local Fallback)**: The default setup where developers connect to a local PostgreSQL instance (e.g., local Supabase container). No AWS configuration or internet connection is required.
* **Scenario B (Cloud Connection)**: Deployed environments (or local testing against cloud databases) fetch credentials dynamically from AWS Secrets Manager using Spring Cloud AWS.

---

## 2. Setup Guides

### Scenario A: Local Development (Default)
To run the API against your local database instance:
1. Start your local database container (default port: `54322`).
2. Run the application normally:
   ```bash
   ./mvnw spring-boot:run
   ```
No Spring profile configuration or AWS credentials are required. The application will automatically fall back to the default credentials:
* **JDBC URL**: `jdbc:postgresql://localhost:54322/postgres`
* **Username/Password**: `postgres` / `postgres`

---

### Scenario B: Connecting to Cloud Database Locally
To run the API locally but connect directly to a cloud database environment (e.g., `staging` or `dev` Supabase database):

1. **Configure AWS CLI**: Ensure your local machine has AWS credentials with permission to read secrets from AWS Secrets Manager:
   ```bash
   aws configure
   ```
2. **Launch with Profiles**: Start the application by enabling both the `aws` profile (which toggles Secrets Manager loading) and the target environment profile (e.g., `staging` or `dev`):
   ```bash
   ./mvnw spring-boot:run -Dspring-boot.run.profiles=aws,staging
   ```

Spring Boot will authenticate using your local AWS CLI credentials, download the staging database connection values from AWS Secrets Manager, and establish the connection.

---

## 3. How it Works Under the Hood

The application's `application.yaml` is split into two logical blocks using the multi-document separator (`---`).

1. **Default Document (Global)**: Defines the fallback datasource parameters using `localhost` defaults.
2. **AWS Profile Document**: Activates **only** when the `aws` profile is requested:
   ```yaml
   ---
   spring:
     config:
       activate:
         on-profile: aws
       import: "aws-secretsmanager:/config/genlab-launchpad-lms-api_${spring.profiles.active:dev}"
   ```

When the `aws` profile is active:
* The application imports settings from AWS Secrets Manager at the path `/config/genlab-launchpad-lms-api_<active-profile>`.
* The key-value pairs stored in the secret are injected directly into Spring's environment properties, overriding the default local database parameters.

---

## 4. Secret Configuration Schema in AWS

Secrets inside AWS Secrets Manager must be stored as **JSON Key-Value Pairs** under the name `/config/genlab-launchpad-lms-api_<env>` (e.g. `/config/genlab-launchpad-lms-api_staging`).

The following keys must be present in the JSON:
```json
{
  "spring.datasource.url": "jdbc:postgresql://db.xxxx.supabase.co:5432/postgres",
  "spring.datasource.username": "postgres",
  "spring.datasource.password": "your-db-password",
  "supabase.url": "https://xxxx.supabase.co",
  "supabase.anon-key": "your-supabase-anon-key",
  "supabase.jwt-secret": "your-supabase-jwt-secret",
  "supabase.service-role-key": "your-supabase-service-role-key"
}
```

## 5. Locating OTPs / Magic Links during testing

When testing sign-in and authentication flows, here is where you can find the One-Time Passwords (OTPs) and magic link emails:

### In Local Development
All outgoing emails generated by the local Supabase Auth (GoTrue) container are captured locally. They are **not** sent to real email addresses.
* **Email Catcher (Inbucket)**: Open your browser and go to **`http://localhost:54324`**.
* Here you will see a list of all captured emails. Click on your verification email to retrieve the 6-digit OTP code or the magic link URL.

### In Deployed Cloud Environments (`dev`, `staging`, `prod`)
* **Email OTP / Magic Links**: These are routed through the configured SMTP provider (**Resend**) and will be delivered to the actual inbox of the email address used for sign-in. You can also view logs and delivery statuses in the organization's **Resend console**.
* **SMS OTP**: Handled and sent **directly by Supabase** (using the SMS provider like Twilio or MessageBird configured in your Supabase project dashboard) to the recipient's mobile phone number.
